# Portal Sensor Explorer — Development Guide

A native Android sensor explorer app for the Meta Portal (1st generation) device. Probes all available hardware: cameras, microphones, sensors, Bluetooth, and network interfaces. Built on top of the [meta-quest/portal-samples](https://github.com/meta-quest/portal-samples) project.

### Build & Deploy Commands

```powershell
# Build
.\gradlew.bat assembleDebug

# Install on Portal (device must be connected via USB-C with ADB enabled)
adb install -r app\build\outputs\apk\debug\app-debug.apk

# Launch
adb shell am start -n net.adrianba.portal.sensorexplorer/net.adrianba.portal.sensorexplorer.MainActivity

# Force stop
adb shell am force-stop net.adrianba.portal.sensorexplorer

# Read app logs
adb logcat -d -s SensorExplorer:I

# Clear logcat before a fresh run
adb logcat -c
```

### Known Build Issues

1. **OneDrive file locking:** If the project is stored on a OneDrive-synced path, file locking can cause `Unable to delete directory` errors during builds. Move the project outside OneDrive or run `Remove-Item -Recurse -Force` on the build directory before rebuilding.

2. **Compose compiler forbids try-catch around @Composable calls.** You cannot wrap composable function calls in try-catch blocks. Extract values into local variables before the composable call, then pass the values in.

3. **Hidden Android APIs:** `android.os.SystemProperties` is a hidden API. Access it via reflection:
   ```kotlin
   val clazz = Class.forName("android.os.SystemProperties")
   val method = clazz.getMethod("get", String::class.java, String::class.java)
   method.invoke(null, key, "") as String
   ```

4. **`NetworkInterface.getNetworkInterfaces()`** returns null on Portal's older AOSP build. Use a fallback that parses output from `Runtime.exec("ip addr")`.

5. **`MicrophoneInfo` constants** like `MICROPHONE_TYPE_BUILTIN` don't exist as named constants in the SDK even though API 28 docs list them. Use raw int values (e.g., `15` for built-in mic type, `1` for location bottom).

6. **`MicrophoneInfo` sentinel values:** The API uses `-Float.MAX_VALUE` (approximately `-3.4e38`) as a sentinel for unknown values, not `Float.NEGATIVE_INFINITY`. Check with `value <= -Float.MAX_VALUE / 2` and display "N/A".

7. **`MicrophoneInfo.frequencyResponse`** returns `List<Pair<Float, Float>>` and `channelMapping` returns `List<Pair<Int, Int>>`.

## Architecture

Single-activity Jetpack Compose app. `MainActivity` extends `ComponentActivity`. All UI is declarative Compose inside `setContent { }`. The `ShowcaseScreen` composable renders a vertically scrollable column of hardware exploration sections separated by dividers.

`MainActivity.onCreate()` also calls `dumpSensorInfo()` which logs comprehensive hardware information to logcat under the tag `SensorExplorer` for easy retrieval via `adb logcat -s SensorExplorer:I`.

### Camera

Single front-facing camera (Camera ID 0):
- Sensor size: 6.3×4.7 mm
- Focal length: 2.4 mm
- Max digital zoom: 8.0×
- Hardware level: 0 (LIMITED)
- JPEG output sizes: 1280×720, 320×240, 640×480
- No rear camera, no external camera

### Microphones

4 built-in bottom-mounted microphones in a far-field array (one group, indices 0–3):
- Type: Built-in (type 15)
- Location: Bottom (location 1)
- Directionality: Omnidirectional (directionality 1)
- Sensitivity, SPL: Not reported (sentinel values)
- Position: Not reported
- Orientation: (0, 0, 0)
- Supported sample rates: 8000, 11025, 12000, 16000, 22050, 24000, 32000, 44100, 48000 Hz
- Channel counts: 1, 2
- Encodings: PCM 16-bit (encoding 2)

Two audio input device IDs are reported (ID 13 and ID 15). Both appear identical in the API. Based on audio policy config, ID 13 maps to all 4 physical microphones ("Built-In Mic"), and ID 15 is likely a virtual/processed input for the back mic routing ("Built-In Back Mic").

### Speaker

- Built-in stereo speaker (output ID 2, type 2)
- Sample rate: 48000 Hz
- Channel count: 2

### Sensors

Only **1 hardware sensor** is present:
- **al3010 Ambient Light Sensor** (type 5 / `TYPE_LIGHT`)
  - Vendor: (empty)
  - Version: 1
  - Max range: 0.0 (not specified)
  - Resolution: 0.0186 lux
  - Power: 0.001 mA
  - Min delay: 200,000 µs (5 Hz max)
  - Max delay: 10,000,000 µs (0.1 Hz min)

  - Wake-up: false

### What Is NOT Present

The Portal is a stationary smart display. The following hardware is **not available**:

| Missing Hardware | Why |
|---|---|
| Accelerometer | Stationary device — no motion sensing needed |
| Gyroscope | No IMU hardware at all |
| Magnetometer / Compass | No orientation sensing |
| Proximity sensor | Not included |
| Barometer / Pressure sensor | Not included |
| Step counter / detector | Not included |
| GPS / GNSS | No location hardware |
| Rear camera | Only front-facing camera |
| NFC | Not included |

The Android framework's 9-axis fusion, game fusion, and geomag fusion modules exist in the system but are disabled (0 clients) due to the absence of IMU hardware.

### Bluetooth

- Adapter name: "Gen1 Portal"
- Real MAC address: visible via `adb shell settings get secure bluetooth_address` or `dumpsys bluetooth_manager`
- API-reported address: `02:00:00:00:00:00` (privacy-masked since Android 8 for non-system apps; `Settings.Secure.getString()` also returns null from app context)
- State: ON, Scan mode: Connectable
- BLE Multi-Advertisement: Supported (Portal can advertise itself)
- BLE Offloaded Filtering: Supported
- BLE Offloaded Scan Batching: Supported
- **BLE Scanning: NOT FUNCTIONAL** — `mMaxScanFilters: 0` in the GATT service means the BT firmware cannot allocate a scanner for BLE scan operations. `startScan()` always fails with `SCAN_FAILED_INTERNAL_ERROR` (error code 3). Only classic Bluetooth discovery works.
- BT library: `libbluetooth_qti.so` (Qualcomm)
- Max connected audio devices: 2
- Available profiles: A2DP Sink, HID Host, AVRCP Controller, GATT Service
- No bonded devices by default

### Network

- WiFi: 5 GHz capable, 802.11ac (866 Mbps link speed observed)
- Network interfaces (via `ip addr` parsing): lo, dummy0, wlan0
- `NetworkInterface.getNetworkInterfaces()` returns null — must use `ip addr` command fallback
- No Ethernet port
- No cellular radio

### User Presence Detection

The Portal has a built-in camera-based presence detection system managed by `aloha.UserPresenceManager` (part of `com.facebook.alohaservices.alohausers`). It detects whether a user is in front of the device and transitions between states.

#### Presence States

| State | Meaning |
|---|---|
| **ACTIVE** | User detected directly in front of the Portal |
| **AMBIENT** | Motion or presence detected nearby |
| **STANDBY** | No user detected — device idle/sleeping |

#### Key Log Messages (tag: `aloha.UserPresenceManager`)

```
COMMAND_DEVICE_PRESENCE_MODE_CHANGE [STANDBY]    # State transition
COMMAND_DEVICE_PRESENCE_MODE_CHANGE [AMBIENT]    # State transition
COMMAND_DEVICE_PRESENCE_MODE_CHANGE [ACTIVE]     # State transition
updateAvailability Availability idle in [STANDBY] mode
maybeResetAccountStateToUnidentified, deviceState [AMBIENT], isSuperframeRunning [true]
SuperFrame state changed: android.intent.action.DREAMING_STARTED   # Screen saver started
SuperFrame state changed: android.intent.action.DREAMING_STOPPED   # Screen saver stopped
COMMAND_TOP_APP_UPATED                           # Foreground app changed
COMMAND_SUPERFRAME_STATE_CHANGE                  # Screen saver state change
COMMAND_LOAD_OWNERS                              # Owner list refresh (periodic)
Owners changed notification                      # Periodic (every 30 min)
```

#### Accessing Presence from a Third-Party App

The presence system APIs are locked behind `signature|privileged` permissions. The only practical approaches for third-party apps are:

| Method | How | Limitation |
|---|---|---|
| **Logcat monitoring** (recommended) | Grant `READ_LOGS` via ADB, read logcat for `aloha.UserPresenceManager` tag, parse `COMMAND_DEVICE_PRESENCE_MODE_CHANGE [STATE]` | Requires `adb shell pm grant <pkg> android.permission.READ_LOGS`; must declare permission in manifest |
| **Screen on/off broadcasts** | Register for `ACTION_SCREEN_ON` / `ACTION_SCREEN_OFF` | Only a proxy — doesn't distinguish AMBIENT from ACTIVE |
| **`ACTION_USER_PRESENT`** | Standard Android broadcast on device wake | Coarse — only fires once on unlock |
| **Own camera-based detection** | Use Camera2 + face detection | Duplicates what the system already does |

#### Locked APIs (NOT accessible to third-party apps)

| API | Permission Required | Protection Level |
|---|---|---|
| `platform_state_service` (AIDL) | `com.facebook.alohasdk.permission.PLATFORM_STATE_SERVICE` | `signature` |
| `UserPresenceService` binding | `com.facebook.aloha.permission.USER_PRESENCE_SERVICE` | `signature\|privileged` |
| `GlobalStatesContentProvider` | `com.facebook.aloha.permission.ACCESS_STATESDB` | `signature\|privileged` |
| Presence transition broadcasts | `com.facebook.aloha.permission.RECEIVE_PRESENCE_TRANSITION` | `signature\|privileged` |
| Send presence transitions | `com.facebook.aloha.permission.SEND_PRESENCE_TRANSITION` | `signature\|privileged` |

Note: While the AIDL `platform_state_service` is locked, `dumpsys platform_state_service` works from app context without special permissions. See "Platform State Bundle" section below.

#### Available Normal Permissions (auto-grantable)

| Permission | Purpose |
|---|---|
| `com.facebook.portal.permission.ACCESS_CAMERA_MIC_STATE` | Query camera/mic shutter state |
| `com.facebook.aloha.permission.RECEIVE_PRIVACY_EVENTS` | Receive privacy state change broadcasts |

#### Implementation Notes

- The app's `PresenceMonitorSection.kt` implements logcat-based monitoring
- `READ_LOGS` must be declared in `AndroidManifest.xml` AND granted via `adb shell pm grant`
- The logcat stream is read on a background thread using `Runtime.exec("logcat -s aloha.UserPresenceManager:I")`
- Historical events are seeded from the logcat buffer on start (`logcat -d`)
- State transitions happen when: user walks up, walks away, or the Portal times out to standby
- "SuperFrame" is the Portal's screen saver / ambient display mode
- The `COMMAND_LOAD_OWNERS` / `Owners changed notification` messages are periodic (every ~30 min) and not presence-related

### Weather Data

The Portal fetches weather periodically. The last fetch is available in the logcat buffer under tag `aloha.WeatherFetcher`:
```
weatherInfo=WeatherInfo(temperature=57, scaleString=FAHRENHEIT, conditionString=mostly_cloudy)
```

The `WeatherSection.kt` parses this from `logcat -d -s aloha.WeatherFetcher:I`.

### Audio Recording Constraints

- **Hey Portal (`com.millennium`)** holds the microphone open at all times for hotword detection
- When `com.millennium` is recording, other apps get silence or minimal audio data (448ms from a 5-second recording)
- **Fix:** Stop Hey Portal before recording: `adb shell am force-stop com.millennium`
- The app attempts `AUDIOFOCUS_GAIN_TRANSIENT_EXCLUSIVE` and `am force-stop com.millennium` but the latter requires system UID
- Recording format: AAC/MPEG4 (`.m4a`), 16kHz, mono, 96kbps
- Audio source: `VOICE_RECOGNITION` (better for far-field mic array than `MIC`)
- The device caps encoding bitrate at 96kbps (128kbps request is silently reduced)

## Platform Constraints (Portal General)

- **No Google Mobile Services (GMS):** Maps, Sign-In, Push, In-App Billing, Firebase will not function. Use non-GMS alternatives. GMS downloadable fonts will silently fall back to system default — bundle font files as TTF assets in `res/font/` instead.
- **SDK versions:** The Portal 1st gen runs Android 9 (SDK 28). Set `minSdkVersion` to 28 and `targetSdkVersion` to 29 for maximum compatibility.
- **Contacts/Credentials:** Access to device contacts and device credentials (e.g. Facebook) is not possible.
- **Storage delete:** Deleting files from the device requires `adb shell rm`; there is no API for this.
- **No GMS location services:** Use `LocationManager` directly if location is needed, but no GPS hardware exists on Portal.
- **Microphone contention:** The `com.millennium` (Hey Portal) process holds the mic permanently for hotword detection. Other apps get silence unless millennium is stopped via `adb shell am force-stop com.millennium`.
- **System overlay:** The Portal system overlay (back, home, Wi-Fi icons) sits ON TOP of apps with no automatic inset. Reserve the top 64dp of screen space.

## Debugging Tips

1. **Logcat:** The app dumps all hardware info on startup under tag `SensorExplorer`. Use `adb logcat -d -s SensorExplorer:I` to read.
2. **Clear logs first:** Run `adb logcat -c` before launching to get a clean dump.
3. **Audio recorder logs:** `adb logcat -d -s AudioRecorder:I` — shows file size, duration, start/stop events.
4. **Shell exploration:** Use `adb shell` to explore device hardware:
   - `dumpsys sensorservice` — sensor details
   - `dumpsys bluetooth_manager` — BT state, profiles, real MAC address
   - `dumpsys media.audio_flinger` — audio hardware details, active recording clients
   - `dumpsys platform_state_service` — full platform state bundle
   - `cat /proc/cpuinfo` — CPU details
   - `cat /sys/devices/soc0/machine` — SoC model
   - `ip addr` — network interfaces
5. **Permissions:** On Android 9, runtime permissions are auto-granted for debug builds when declared in the manifest. No need for runtime permission requests for the sensor explorer sections.
6. **Aloha services logcat:** `adb logcat -s aloha.UserPresenceManager:I aloha.WeatherFetcher:I aloha.DeviceStateHelper:I aloha.ResourceProfileManager:I`
7. **Mic contention:** If audio recording captures <1s of audio, `com.millennium` is holding the mic. Stop it with `adb shell am force-stop com.millennium`.
8. **Live logcat monitoring:** `adb logcat` streams in real-time. Use `adb logcat -s TAG:LEVEL` to filter by tag.
